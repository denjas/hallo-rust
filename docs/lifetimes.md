# Времена жизни (lifetimes)

> Ступень 3 · Урок C, закрывает ступень. Опирается на [`borrowing.md`](./borrowing.md) и [`ownership.md`](./ownership.md).

В `borrowing.md` мы упёрлись в правило «ссылка обязана быть валидной» и в запрет висячих ссылок. Времена жизни — это аппарат, которым компилятор **доказывает**, что ни одна ссылка не переживёт данные, на которые смотрит. Это не новая сущность в рантайме, а способ записать в типах ответ на вопрос: «как долго эта ссылка обязана оставаться годной?»

Сразу сними главный страх: **времена жизни почти всегда выводятся автоматически, и писать их руками приходится редко.** Большую часть времени ты их даже не видишь. Этот урок — про то, что они есть, что значат и когда всё-таки придётся проставить их вручную.

---

## 1. Что такое время жизни

Время жизни (lifetime) — это **участок кода, на котором ссылка обязана оставаться валидной**. У каждой ссылки оно есть всегда, просто обычно неявно. Borrow checker сопоставляет два срока: сколько живёт значение и сколько живёт ссылка на него. Если ссылка способна пережить значение — это ошибка.

```rust
{
    let r;                      // r — ссылка
    {
        let x = 5;
        r = &x;                 // r смотрит на x
    }                           // x уничтожается здесь
    // println!("{r}");         // ОШИБКА: r пережил бы x → висячая ссылка
}
```

```text
error[E0597]: `x` does not live long enough
```

Логика компилятора дословно: `x` живёт только до своей `}`, а `r` пытаются использовать после неё — значит ссылка переживёт данные, запрещено. **Никакого рантайма тут нет** — это чистая проверка сроков на этапе компиляции. В C++ ровно этот код дал бы висячую ссылку и UB, причём молча.

---

## 2. Чаще всего ты их не пишешь (элизия)

Компилятор сам расставляет времена жизни в типичных случаях — это называется **элизия** (lifetime elision). Поэтому вот это пишется без единой аннотации:

```rust
fn first_char(s: &str) -> Option<char> {   // никаких 'a — и так понятно
    s.chars().next()
}
```

Компилятор рассуждает так: «возвращаемая ссылка может происходить только из единственного входного `&str`, значит её срок привязан к нему». Когда вывод однозначен, аннотации не нужны. Большинство функций со ссылками попадают сюда — оттого `'a` так редко мелькает в обычном коде.

---

## 3. Когда приходится писать `'a` руками

Аппарат вылезает наружу, когда из **нескольких** входных ссылок компилятор не может сам понять, к какой привязан результат. Классика — функция, возвращающая одну из двух ссылок:

```rust
fn longest(a: &str, b: &str) -> &str {     // ОШИБКА: к чьему сроку привязан результат?
    if a.len() >= b.len() { a } else { b }
}
```

```text
error[E0106]: missing lifetime specifier
```

Компилятор не угадывает: результат — это `a` или `b`? У них могут быть разные сроки жизни. Надо ему **сказать**, и для этого вводят именованное время жизни `'a`:

```rust
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}
```

Читается так: «есть некий срок `'a`; оба аргумента живут **хотя бы** `'a`, и результат годен **тоже** `'a`». Это не меняет поведение программы и ничего не стоит в рантайме — это **обещание компилятору**, которое он затем проверяет на каждом вызове.

Синтаксис: `'a` — имя времени жизни (апостроф + строчное имя, по конвенции короткое — `'a`, `'b`). `<'a>` после имени функции объявляет его, как `<T>` объявляет обобщённый тип. Дальше `&'a str` означает «ссылка со временем жизни `'a`».

### Как это работает на стороне вызова

`'a` — не фиксированный отрезок, а «дырка», которую компилятор на каждом вызове заполняет **пересечением** реальных сроков аргументов. Поэтому он ловит ошибку, если результат переживёт более короткоживущий аргумент:

```rust
let s1 = String::from("длинная");
let result;
{
    let s2 = String::from("кор");
    result = longest(&s1, &s2);   // 'a сжимается до срока s2 (он короче)
    println!("{result}");         // ок: пока s2 жива
}                                 // s2 умирает
// println!("{result}");          // ОШИБКА: result был годен лишь пока жива s2
```

Аннотация `'a` дала компилятору связь «результат не переживёт самый короткий из входов», и он её применил.

## 4. Времена жизни в структурах

Если структура **хранит ссылку** (а не владеющее значение), ей тоже нужно время жизни — обещание «структура не переживёт того, на что ссылается»:

```rust
struct Excerpt<'a> {
    text: &'a str,        // структура держит заимствованный кусок чужой строки
}

fn main() {
    let novel = String::from("Был ясный холодный день.");
    let first = novel.split('.').next().unwrap();
    let e = Excerpt { text: first };   // e заимствует часть novel
    println!("{}", e.text);
}   // компилятор следит, чтобы e не пережила novel
```

`<'a>` у структуры говорит: «экземпляр `Excerpt` живёт не дольше, чем строка, на которую смотрит его поле `text`». Это та же идея, что в функции, перенесённая на тип. (Структура из владеющих полей, как `String`, времени жизни не требует — она ни от чего не заимствует.)

### Когда сроков несколько: они должны быть независимы

Важно не плодить имена сроков ради того, что «ссылок несколько». **Сам факт двух ссылок не требует двух имён** — лишнее имя это просто шум. Разные сроки нужны лишь там, где две ссылки реально **живут независимо**, и общий срок насильно сцепил бы их жизни.

Это почти всегда про структуры, а не про функции (у функций, как правило, либо хватает одного `'a`, либо всё выводит элизия). Канонический случай — тип, держащий две ссылки, которые живут по-разному:

```rust
struct Parser<'src, 'cfg> {
    input: &'src str,     // ссылка на разбираемый текст
    config: &'cfg str,    // ссылка на конфиг — живёт совершенно иначе
}
```

Зачем здесь два срока, а не один общий? Потому что текст и конфиг **по жизни не связаны**: один конфиг может пережить много разных входных текстов, а каждый отдельный текст — умереть задолго до конфига. С двумя сроками компилятор позволяет `input` умирать раньше `config` (и наоборот), не цепляя их друг к другу:

```rust
let config = String::from("режим=строгий");   // долгоживущий конфиг
{
    let input = String::from("данные на разбор"); // короткоживущий текст
    let p = Parser { input: &input, config: &config };
    // ... используем p ...
}                                              // input умирает здесь — это ОК
// config всё ещё жив и может пойти в следующий Parser с другим input
```

Будь у обоих полей один общий `'a`, компилятор приравнял бы их сроки и потребовал, чтобы `config` и `input` жили одинаково долго — а это неправда и сделало бы тип неудобным или непригодным. Раздельные `'src` и `'cfg` снимают эту ложную связь.

**Правило в карман:** одно имя срока на несколько ссылок = «их сроки связаны»; разные имена = «сроки независимы». Ставь столько имён, сколько реально **независимых** связей, не больше.

## 5. Зачем всё это: что выводится из модели

Свяжем три урока ступени в одну картину. Владение, заимствование и времена жизни — это один механизм с трёх сторон, и вместе они дают гарантию, которой нет ни у C++, ни у GC-языков:

1. **Память безопасна без сборщика мусора.** Нет use-after-free, double-free, висячих ссылок — всё это невыразимо, а не «обычно не случается». Цена в рантайме — ноль.
2. **Освобождение детерминировано.** Ты знаешь точный момент `Drop` (выход из скоупа), без пауз GC — важно для эмбеда и реалтайма.
3. **Гонки данных невозможны на этапе компиляции.** Правило «разделение XOR изменение» из `borrowing.md` плюс контроль сроков — фундамент «бесстрашной многопоточности» (Ступень 7).

Времена жизни — последний кусок: они гарантируют, что заём (`borrowing.md`) никогда не переживёт владельца (`ownership.md`). Borrow checker — это и есть совместная проверка всех трёх.

---

## Упражнение

1. Воспроизведи ошибку `does not live long enough`: заведи внешнюю `let r;`, во вложенном блоке создай `x` и сделай `r = &x;`, потом используй `r` снаружи блока. Прочитай ошибку, объясни её через сроки.
2. Напиши `longest` сначала **без** `'a` — поймай `missing lifetime specifier`. Потом добавь `<'a>` и `&'a str`, добейся компиляции.
3. Сломай его осознанно: вызови `longest` так, чтобы результат использовался после смерти более короткоживущего аргумента (как в §3). Убедись, что компилятор это ловит, и проговори, что именно он проверил.
4. Объяви структуру `Excerpt<'a> { text: &'a str }`, создай экземпляр, заимствующий часть `String`. Затем попробуй уронить `String` раньше структуры — поймай ошибку.

## Шпаргалка

- **Время жизни** = участок, на котором ссылка обязана быть валидной. Есть у каждой ссылки, обычно неявно. Чистая проверка на этапе компиляции, в рантайме её нет.
- **Элизия:** в типичных случаях (одна входная ссылка) компилятор выводит сроки сам — аннотации не нужны.
- **`'a` пишут руками**, когда из нескольких входных ссылок неясно, к чьему сроку привязан результат (`fn longest<'a>(a: &'a str, b: &'a str) -> &'a str`).
- `'a` — это **обещание** компилятору «эти ссылки соотносятся по срокам так-то»; он проверяет его на каждом вызове, подставляя пересечение реальных сроков.
- **Структура со ссылкой** получает `<'a>`: «не переживу того, на что ссылаюсь». Структура из владеющих полей `'a` не требует.
- Вместе с владением и заимствованием времена жизни гарантируют: **заём никогда не переживёт владельца**. Это и есть borrow checker.
