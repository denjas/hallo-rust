# Enum'ы: типы-суммы, `Option`, `Result`

> Ступень 2 · Урок A. Парный файл: [`match.md`](./match.md) — про разбор этих значений.

Главная идея, ради которой всё затевалось, — **сделать нелегальные состояния невыразимыми** (make illegal states unrepresentable). В C++ и Python ты постоянно держишь в голове инварианты вроде «это поле валидно, только если тот флаг `true`». Они живут в голове и в комментариях — компилятор о них не знает. Rust даёт инструмент засунуть их в систему типов, чтобы невалидное состояние просто нельзя было собрать. Enum'ы — главный такой инструмент.

---

## 1. Enum — это не то, что ты думаешь

В C `enum` — это переодетый `int`: набор именованных констант, который можно сложить, сравнить с числом и прострелить себе ногу. В Rust enum — это **тип-сумма** (он же алгебраический тип): значение — это *ровно один* из вариантов, и компилятор это сторожит.

Простейший случай выглядит как в C, но под капотом это другое:

```rust
enum Direction {
    North,
    South,
    East,
    West,
}
```

Уже выигрыш: `Direction` нельзя сравнить с числом и нельзя случайно получить пятое значение.

## 2. Варианты могут нести данные

Настоящая мощь enum'а в том, что **каждый вариант может нести свой набор данных**:

```rust
enum Shape {
    Circle { radius: f64 },                  // вариант с именованными полями
    Rectangle { width: f64, height: f64 },   // другой набор полей
    Triangle(f64, f64, f64),                 // кортежный вариант (поля без имён)
    Point,                                    // вариант вообще без данных
}
```

Это `std::variant<Circle, Rectangle, Triangle>` из C++, но нативный, эргономичный и безопасный.

В чистом C ты бы собирал это руками: `struct { enum tag; union {...}; }` — и каждый раз молился, что читаешь то поле `union`, которое соответствует текущему `tag`. Прочитал не то — UB. Rust склеивает тег и доступ к данным намертво: ты **физически не можешь** прочитать поля `Circle`, не доказав компилятору, что значение сейчас именно `Circle`.

### Варианты — это не типы

Важно не запутаться: `Circle`, `Rectangle`, `Triangle`, `Point` — **не типы**. Тип здесь один — `Shape`. Это его *варианты* (формы, которые может принять значение `Shape`), и существуют они только как части `Shape`. Обратиться к варианту можно лишь через имя enum'а: `Shape::Circle`, `Shape::Point`.

Настоящие **типы** в этом объявлении — только то, что стоит в полях: `f64` у всех числовых полей. Всё остальное (`Circle`, `radius`, ...) — имена, которые ты вводишь сам.

Сконструировать значение — значит выбрать вариант и заполнить его поля:

```rust
let a = Shape::Point;                       // форма без данных
let b = Shape::Circle { radius: 5.0 };      // форма с именованным полем
let c = Shape::Triangle(3.0, 4.0, 5.0);     // кортежная форма
```

И `a`, и `b`, и `c` имеют один тип — `Shape`; различаются лишь формой. `Circle`, `Triangle`, `Point` тут работают как **конструкторы** значения `Shape`.

**Формула:** `enum Имя { ... }` вводит один новый тип; имена внутри — его варианты-конструкторы, а не отдельные типы. Реальные типы — только то, что стоит в полях (`f64`, `String`, ...).

### Как отличить определение варианта от ссылки на тип

Когда читаешь чужой enum, бывает неясно: вот это имя — новый вариант или ссылка на тип, объявленный где-то раньше? Регистр не поможет — и типы, и варианты пишутся с большой буквы. Помогает **позиция**.

В теле `enum { ... }` и `struct { ... }` существующий тип может появиться только в **позиции поля**: внутри `(...)` или после `имя_поля:`. Всё остальное — имена, которые ты вводишь.

```rust
enum Status { Online, Offline }   // отдельный enum, объявлен заранее

enum Link {
    Disconnected,          // голое имя, НЕ в позиции поля → НОВЫЙ вариант
    Down(Status),          // Status в скобках = позиция поля → ССЫЛКА на тип Status
    Up { id: String },     // String после id: = позиция поля → ССЫЛКА на тип String
}
```

`Disconnected` стоит голым (после него нет ни `(`, ни `{`, ни `:`) → по грамматике это может быть только определение нового unit-варианта. А `Status` и `String` стоят в позициях полей → это ссылки на существующие типы.

Голым именем существующий тип в теле enum'а оказаться не может — для этого нет синтаксиса. Enum'ы не наследуются и не «включают» друг друга по имени; чужой тип входит только как тип поля.

Если сомнения остались — посмотри на использование: вариант конструируется и матчится через имя своего enum'а (`Link::Disconnected`), а самостоятельный тип — через своё имя (`Status::Online`). А `rust-analyzer` на ховере прямо подпишет, что это вариант.

### Вариант может нести другой enum

Полем варианта может быть и другой enum — как `Down(Status)` выше. Полный цикл: объявление → создание → разбор.

**Создание.** Раз `Down` несёт `Status`, при конструировании кладём внутрь конкретный `Status`:

```rust
let a = Link::Down(Status::Offline);             // Status::Offline становится начинкой Down
let b = Link::Up { id: String::from("sess-42") };
```

`Status::Offline` — значение типа `Status`; оно становится начинкой варианта `Down`. Тип `a` при этом — `Link` (а не `Status`); `Status` сидит внутри.

**Разбор — через `match`** (механику смотри в [`match.md`](./match.md)). Паттерн `Down(status)` привязывает начинку к переменной, дальше разбираем её:

```rust
fn describe(link: &Link) {
    match link {
        Link::Disconnected => println!("отключено"),
        Link::Down(status) => match status {     // status: &Status — то, что лежало внутри Down
            Status::Online  => println!("помечено Down, но статус online"),
            Status::Offline => println!("закрыто, причина: offline"),
        },
        Link::Up { id } => println!("активно, сессия {}", id),
    }
}
```

Вложенные `match` схлопываются — паттерн умеет заглядывать вглубь за один шаг:

```rust
match link {
    Link::Disconnected          => println!("отключено"),
    Link::Down(Status::Online)  => println!("помечено Down, но статус online"),
    Link::Down(Status::Offline) => println!("закрыто, причина: offline"),
    Link::Up { id }             => println!("активно, сессия {}", id),
}
```

Бонус: исчерпывающесть проверяется **по всему дереву**. Добавишь в `Status` третий вариант (`Connecting`) — этот `match` перестанет компилироваться, пока не разберёшь `Down(Status::Connecting)`. Бесплатное напоминание про новое вложенное состояние; в C++ с `std::variant` + вложенными `switch` такого сторожа нет.

## 3. `Option<T>` — казнь null'а

В Rust **нет null. Совсем.** Ни в каком виде. Ссылка `&T` всегда указывает на живое валидное значение, точка. Это убирает целый класс багов, который Тони Хоар (изобретатель null'а) назвал «ошибкой на миллиард долларов».

Как тогда выразить «значение может отсутствовать»? Обычным enum'ом из стандартной библиотеки:

```rust
enum Option<T> {
    Some(T),    // значение есть, вот оно внутри
    None,       // значения нет
}
```

Никакой магии — это просто enum с дженериком `T` (про дженерики подробно на Ступени 4; пока читай `<T>` как «для любого типа»). Гениальность в том, что **отсутствие стало частью типа** и видно в сигнатуре:

```rust
fn first_char(s: &str) -> Option<char> {
    s.chars().next()        // None для пустой строки, Some(c) иначе
}
```

Чтобы добраться до значения внутри `Some`, ты **обязан** разобрать `None` — компилятор не даст притвориться, что отсутствия не бывает. Способы разбора (`match`, `if let`, `let ... else`) — в [`match.md`](./match.md).

Сравни: в Python `None` просочится сквозь пять функций и рванёт `AttributeError` где-то совсем не там; в C++ разыменование `nullptr` даст segfault или тихое UB. Rust заставляет разобраться с пустотой *в точке, где она возникла*. Цену платишь компилятору, а не клиенту в проде в три ночи.

## 4. `Result<T, E>` — ошибки как значения, без исключений

В Rust **нет исключений** в C++/Python смысле. Ошибки — это обычные возвращаемые значения, и снова через enum:

```rust
enum Result<T, E> {
    Ok(T),     // успех, внутри полезное значение
    Err(E),    // провал, внутри информация об ошибке
}
```

Функция, способная упасть, возвращает `Result` — и это написано в её типе, проглядеть нельзя:

```rust
fn parse_age(s: &str) -> Result<u32, std::num::ParseIntError> {
    s.parse::<u32>()        // .parse сам возвращает Result
}
```

Философская разница с исключениями: исключение — это **невидимый поток управления**. Глядя на `foo()` в C++, ты не знаешь, выпрыгнет ли оно стеком вверх через твою голову. В Rust `Result` делает возможность провала *видимой в типе и в коде*.

### Оператор `?` — клей, который делает всё это терпимым

Минус `Result` — многословность, если разбирать каждый руками. Лекарство — оператор `?`:

```rust
fn doubled_age(s: &str) -> Result<u32, std::num::ParseIntError> {
    let n = s.parse::<u32>()?;   // вот этот ?
    Ok(n * 2)
}
```

`?` после выражения типа `Result` означает: «если внутри `Ok` — распакуй значение и продолжай; если `Err` — немедленно верни этот `Err` из всей функции». Одна строка заменяет вот это:

```rust
let n = match s.parse::<u32>() {
    Ok(v) => v,
    Err(e) => return Err(e),     // ровно это и прячет ?
};
```

`?` работает и с `Option` (там он на `None` делает ранний `return None`).

Сравни три мира: в Go ты пишешь `if err != nil { return err }` после каждого вызова — ритуал. В C++ исключения экономят буквы, но прячут поток управления. Rust держит баланс: провал **виден** (вот он, `?`, прямо в строке), но не **многословен**. Это, пожалуй, лучшее решение по обработке ошибок среди мейнстрим-языков — и оно возможно именно потому, что ошибка — это просто значение enum'а.

Деталь: `Result` помечен `#[must_use]`. Проигнорируешь возвращённый `Result` — компилятор ругнётся предупреждением. Молча уронить ошибку на пол, как сплошь и рядом в C, тут незаметно не выйдет.

## 5. Где всё сходится: проектирование типами

Вернёмся к «нелегальные состояния невыразимы». Пусть есть соединение: либо подключено (и тогда есть сессия), либо нет (и сессии нет).

Наивный C++/Python-подход — отдельный флаг и опциональное поле:

```rust
// ПЛОХО: можно собрать бессмыслицу
struct Connection {
    is_connected: bool,
    session_id: Option<String>,
    // что значит is_connected = true, но session_id = None?
    // что значит is_connected = false, но session_id = Some(...)?
    // две невалидных дырки, в которые рано или поздно провалится баг
}
```

Здесь четыре комбинации полей, а осмысленных — две. Rust-подход — закодировать инвариант в тип так, чтобы дырок не было физически:

```rust
// ХОРОШО: невалидное состояние просто не собрать
enum Connection {
    Disconnected,
    Connected { session_id: String },
}
```

Теперь «подключён без сессии» и «не подключён, но с сессией» — не баги, которые надо тестировать, а состояния, которые **невозможно сконструировать**. Покажем это на конструировании:

```rust
let off = Connection::Disconnected;
let on  = Connection::Connected { session_id: String::from("sess-42") };
```

`Connected` без `session_id` собрать невозможно — компилятор потребует поле. А «`Disconnected` со строкой» просто негде выразить: у этого варианта нет полей. Невалидные комбинации не отсеиваются тестами — их физически нет в системе типов.

Это и есть ментальная перестройка, ради которой ты учишь Rust: инварианты переезжают из головы и комментариев в систему типов, и компилятор становится со-проектировщиком.

## Упражнение

`cargo new calculator`. Задействует и enum'ы, и `match` (см. [`match.md`](./match.md)):

1. Объяви enum `Op` с вариантами `Add`, `Sub`, `Mul`, `Div`.
2. Объяви enum `CalcError` с вариантами `DivisionByZero` и `Overflow` (пока без данных).
3. Напиши `fn apply(op: Op, a: f64, b: f64) -> Result<f64, CalcError>`. Для `Div` на ноль возвращай `Err(CalcError::DivisionByZero)`. Используй `match` по `op` — **без** `_`, перечисли все варианты явно.
4. Напиши `fn parse_op(c: char) -> Option<Op>` (`'+' -> Some(Op::Add)` и т.д., прочее `None`).
5. В `main` собери цепочку: распарси символ, обработай `None` через `let ... else`, вызови `apply`, разбери `Result` через `match`. Где сможешь — используй `?` в функции-обёртке, возвращающей `Result`.
6. Проверь себя: добавь в `Op` пятый вариант `Pow` и **не** трогай `apply`. Убедись, что компилятор сам показал, где дописать обработку.

## Шпаргалка

- **enum** = «значение — ровно один из вариантов», варианты могут нести разные данные.
- **варианты — это конструкторы своего enum'а**, а не отдельные типы; реальные типы — только в полях (`f64`, `String`, ...).
- **как отличить вариант от ссылки на тип:** чужой тип стоит только в позиции поля (в `(...)` или после `имя:`); голое имя — это новый вариант.
- **Option<T>** = замена null: `Some(value)` или `None`, отсутствие видно в типе.
- **Result<T, E>** = замена исключений: `Ok(value)` или `Err(error)`, провал видно в типе.
- **`?`** = «распакуй `Ok`/`Some`, иначе ранний возврат `Err`/`None`».
- Цель всего — **нелегальные состояния невыразимы**: инварианты в типах, а не в голове.
