# Умные указатели: `Box`, `Rc`/`Arc`, `RefCell`

> Ступень 6. Опирается на всю Ступень 3 ([`ownership.md`](./ownership.md), [`borrowing.md`](./borrowing.md), [`lifetimes.md`](./lifetimes.md)).

До сих пор у каждого значения был **ровно один владелец**, и заимствование проверялось на этапе компиляции. Это покрывает 90% кода. Но бывают структуры, которые в эту модель «один владелец, проверка статическая» не влезают: дерево, где узел делят несколько родителей; граф; данные, общие между потоками. Умные указатели — это инструменты, чтобы выразить такие случаи, **не теряя безопасности**. Для C++-ника это прямые родственники `unique_ptr`, `shared_ptr` и кое-чего, чему в C++ аналога нет.

---

## 1. `Box<T>` — значение в куче, один владелец

Самый простой. `Box<T>` кладёт значение в кучу, а на стеке остаётся только указатель. Владелец по-прежнему один — это `unique_ptr<T>` из C++.

```rust
let b = Box::new(5);     // 5 лежит в куче, b владеет им
println!("{b}");         // 5 — Box прозрачно разыменовывается
// в конце скоупа b уничтожается, куча освобождается
```

Для простого `i32` это бессмысленно (число и так дёшево). `Box` нужен в двух случаях:

1. **Рекурсивные типы.** Тип, содержащий сам себя, имел бы бесконечный размер на стеке. `Box` разрывает рекурсию: его размер известен (это указатель), а данные — в куче.

```rust
enum List {
    Cons(i32, Box<List>),   // без Box: "рекурсивный тип бесконечного размера" — ошибка
    Nil,
}
```

2. **Трейт-объекты** (`Box<dyn Trait>` из [`traits.md`](./traits.md)) — когда конкретный тип неизвестен и нужен размер указателя.

`Box` — это база: одно владение, просто в куче. Остальные указатели добавляют к этому что-то ещё.

## 2. `Rc<T>` — несколько владельцев (счётчик ссылок)

А если у значения должно быть **несколько владельцев** — например, узел графа, на который ссылаются двое? Правило «один владелец» это запрещает. `Rc<T>` (reference counted) ослабляет его: значение живёт, пока на него есть хоть один владелец, и уничтожается, когда последний исчез. Это `shared_ptr<T>` из C++.

```rust
use std::rc::Rc;

let a = Rc::new(String::from("общая"));   // счётчик владельцев = 1
let b = Rc::clone(&a);                     // ещё один владелец; счётчик = 2 (данные НЕ копируются!)
let c = Rc::clone(&a);                     // счётчик = 3
println!("{}", Rc::strong_count(&a));      // 3
// по мере выхода a, b, c из скоупа счётчик падает; на 0 — String освобождается
```

Ключевое: `Rc::clone` — это **не** глубокая копия данных (в отличие от `.clone()` на `String`). Он лишь копирует указатель и **увеличивает счётчик**. Дёшево. Данные одни на всех.

Цена: счётчик живёт в рантайме, его инкремент/декремент — небольшая, но ненулевая работа (в отличие от обычного владения, которое бесплатно). Поэтому `Rc` берут только когда множественное владение реально нужно, а не «на всякий случай».

Важное ограничение: **`Rc` даёт только разделяемый (`&`) доступ.** Через него нельзя менять данные — иначе нарушилось бы правило «много читателей XOR один писатель» из [`borrowing.md`](./borrowing.md). И вот тут нужен следующий инструмент.

## 3. `RefCell<T>` — проверка заимствования в рантайме

Что, если данные под `Rc` всё-таки надо менять? Или шире: бывает код, который ты-человек знаешь как безопасный, но borrow checker не может это доказать статически. `RefCell<T>` предлагает сделку: **перенести проверку правил заимствования с этапа компиляции на рантайм.**

```rust
use std::cell::RefCell;

let cell = RefCell::new(5);

*cell.borrow_mut() += 1;          // взять &mut в рантайме, изменить
println!("{}", cell.borrow());    // взять &, прочитать → 6
```

`RefCell` — это «внутренняя изменяемость» (interior mutability): снаружи он выглядит неизменяемым, но позволяет менять содержимое через `borrow_mut()`. Правила те же самые (один писатель XOR много читателей), но проверяются **в момент вызова** `borrow`/`borrow_mut`, а не компилятором:

```rust
let cell = RefCell::new(5);
let r1 = cell.borrow_mut();
let r2 = cell.borrow_mut();        // КОМПИЛИРУЕТСЯ, но ПАНИКУЕТ в рантайме:
// thread panicked: already borrowed: BorrowMutError
```

Вот в чём сделка: компилятор такое пропустит, но программа **упадёт в рантайме**, если ты нарушишь правила. Ты обменял гарантию «не соберётся» на гибкость ценой «может паниковать». Цена ещё и в скорости: проверка флагов заимствования в рантайме (хоть и копеечная). В C++ прямого аналога нет — там нарушение правил даёт UB, а `RefCell` хотя бы честно паникует, а не молча портит память.

## 4. Боевая комбинация: `Rc<RefCell<T>>`

По отдельности `Rc` (много владельцев, только чтение) и `RefCell` (изменяемость, рантайм-проверка) половинчаты. Вместе они дают **разделяемые изменяемые данные** — то, ради чего обычно сюда и приходят (узлы графов, деревьев с обратными ссылками, разделяемое состояние):

```rust
use std::rc::Rc;
use std::cell::RefCell;

let shared = Rc::new(RefCell::new(vec![1, 2, 3]));
let clone = Rc::clone(&shared);           // второй владелец того же вектора

clone.borrow_mut().push(4);               // меняем через одного владельца...
println!("{:?}", shared.borrow());        // ...видно через другого: [1, 2, 3, 4]
```

Читается послойно: `Rc` даёт нескольких владельцев, `RefCell` внутри разрешает изменение с рантайм-проверкой. Это рабочая лошадка для связных структур данных в однопоточном коде. Многословно — да, но эта многословность честно отражает реальную сложность («несколько хозяев меняют одно») вместо того, чтобы прятать её, как `shared_ptr` на голую структуру в C++.

## 5. `Arc` — то же, что `Rc`, но для потоков

`Rc` использует **необычный** (нессинхронизированный) счётчик — быстро, но небезопасно между потоками. Для многопоточности есть `Arc<T>` (atomically reference counted): тот же API, но счётчик атомарный. Чуть дороже `Rc`, зато его можно делить между потоками.

```rust
use std::sync::Arc;
let data = Arc::new(vec![1, 2, 3]);
let data2 = Arc::clone(&data);    // можно отправить data2 в другой поток
```

А пара для изменяемости в многопоточье — не `RefCell` (он тоже однопоточный), а `Mutex`/`RwLock`. Связка `Arc<Mutex<T>>` — многопоточный аналог `Rc<RefCell<T>>`. Подробно — на Ступени 7, тут только метка: **`Rc`/`RefCell` — однопоточные, `Arc`/`Mutex` — многопоточные.** Компилятор не даст перепутать (через трейты `Send`/`Sync`).

## 6. Как выбирать: дерево решений

Не хватайся за умный указатель по привычке из C++. Порядок такой:

1. **Обычное владение / `&` / `&mut`** — по умолчанию. Покрывает почти всё, бесплатно, проверено компилятором.
2. **`Box<T>`** — нужно положить в кучу: рекурсивный тип или трейт-объект. Всё ещё один владелец.
3. **`Rc<T>`** — реально нужно несколько владельцев одного значения (однопоточно), только чтение.
4. **`RefCell<T>`** — нужна изменяемость там, где компилятор не может доказать безопасность статически (часто внутри `Rc`).
5. **`Rc<RefCell<T>>`** — разделяемые изменяемые данные (графы, деревья) в одном потоке.
6. **`Arc<Mutex<T>>`** — то же между потоками (Ступень 7).

Каждый шаг вниз добавляет рантайм-стоимость и/или возможность паники. Бери минимально необходимое. Если код просится в `Rc<RefCell<...>>` слишком часто — иногда это сигнал пересмотреть структуру данных (например, хранить узлы в `Vec` и ссылаться индексами), но это уже вопрос архитектуры.

---

## Упражнение

1. Объяви рекурсивный `enum List { Cons(i32, Box<List>), Nil }` и собери список `1 → 2 → 3 → Nil`. Убери `Box` и прочитай ошибку про бесконечный размер.
2. Создай `Rc::new(String)`, сделай два `Rc::clone`, печатай `Rc::strong_count` после каждого. Дай клонам выйти из вложенного скоупа и посмотри, как счётчик падает.
3. Возьми `RefCell::new(0)`, увеличь через `borrow_mut()`. Затем намеренно возьми два `borrow_mut()` одновременно и поймай рантайм-панику `BorrowMutError` — прочувствуй «проверка съехала в рантайм».
4. Собери `Rc<RefCell<Vec<i32>>>`, заведи два владельца, добавь элемент через одного, прочитай через другого. Это и есть разделяемое изменяемое состояние.

## Шпаргалка

- **`Box<T>`** — значение в куче, один владелец (`unique_ptr`). Для рекурсивных типов и трейт-объектов.
- **`Rc<T>`** — несколько владельцев через счётчик ссылок (`shared_ptr`), однопоточно, **только чтение**. `Rc::clone` копирует указатель + инкремент счётчика, не данные.
- **`RefCell<T>`** — внутренняя изменяемость: правила заимствования проверяются **в рантайме** (`borrow`/`borrow_mut`), нарушение → **паника**, не ошибка компиляции.
- **`Rc<RefCell<T>>`** — разделяемые изменяемые данные (графы/деревья) в одном потоке.
- **`Arc<T>`/`Mutex<T>`** — потоковые версии `Rc`/`RefCell` (Ступень 7). `Rc`/`RefCell` — однопоточные.
- **Выбор:** обычное владение → `Box` → `Rc` → `RefCell` → `Rc<RefCell>` → `Arc<Mutex>`. Бери минимально необходимое; каждый шаг — рантайм-цена и/или риск паники.
