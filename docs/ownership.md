# Владение (ownership)

> Ступень 3 · Урок A. Парные файлы: [`borrowing.md`](./borrowing.md), [`lifetimes.md`](./lifetimes.md). Опирается на [`basics.md`](./basics.md) (особенно `String` vs `&str`).

Это сердце Rust. Та дисциплина, которую хороший C++-разработчик держит в голове руками — «кто владеет этим ресурсом? кто его освободит? жив ли он ещё?» — здесь вынесена в систему типов, и компилятор проверяет её за тебя. Цена — нулевая в рантайме: никакого сборщика мусора, никаких счётчиков ссылок по умолчанию. Всё решается на этапе компиляции.

Тут начнётся та самая «драка с borrow checker». Это нормально и проходит. Он не вредничает — он показывает инварианты, которые ты в C++ нарушал молча и иногда успешно.

---

## 1. Какую проблему решает владение

Управление памятью бывает трёх сортов:

1. **Ручное** (C, C++ без RAII): ты сам зовёшь `free`/`delete`. Гибко и быстро, но отсюда весь зоопарк багов — use-after-free, double-free, утечки, висячие указатели.
2. **Сборщик мусора** (Python, Java, Go): рантайм сам находит и освобождает ненужное. Безопасно, но за это платишь паузами GC, лишней памятью и непредсказуемостью момента освобождения.
3. **Владение** (Rust): компилятор на этапе сборки расставляет освобождение за тебя по строгим правилам. Безопасность как у GC, цена как у ручного управления (нулевая в рантайме). Это третий путь, и он — главная инновация языка.

---

## 2. Три правила владения

Вся модель сводится к трём правилам. Заучи их — остальное следствия:

1. **У каждого значения ровно один владелец** (переменная, которая за него отвечает).
2. **Когда владелец выходит из области видимости — значение уничтожается** (освобождается).
3. **Владелец может быть только один в каждый момент.** Передать владение можно (это называется *move*), но тогда у старого владельца его больше нет.

---

## 3. Область видимости и `Drop` (это RAII)

Правило 2 на практике: значение живёт до закрывающей `}` своего блока, и там автоматически уничтожается. Для C++-нику это ровно **RAII** — деструктор по выходу из скоупа.

```rust
{
    let s = String::from("привет");   // s владеет буфером в куче
    // ... s доступна ...
}   // ← здесь s выходит из области видимости: её буфер в куче освобождается автоматически
// s здесь уже не существует
```

`String` устроена как `std::string` в C++: на стеке лежит «шапка» из трёх слов (указатель на буфер в куче, длина, ёмкость), а сами байты — в куче. По выходу из блока Rust сам зовёт деструктор (трейт `Drop`), который освобождает буфер. Никакого `free` руками, и забыть его невозможно.

Если в блоке несколько владеющих значений, они уничтожаются **в обратном порядке объявления** (как локальные объекты в C++):

```rust
{
    let a = String::from("первая");
    let b = String::from("вторая");
}   // сначала уничтожается b, потом a
```

### Shadowing не уничтожает значение раньше времени

Важная тонкость (мы её касались в [`basics.md`](./basics.md)): перекрытие имени через `let` **не** освобождает старое значение в момент перекрытия. Старое значение просто становится недоступным по имени, но физически доживает до конца блока:

```rust
{
    let v = String::from("первая");   // владеет памятью в куче
    let v = String::from("вторая");   // shadowing: "первая" больше не достать по имени...
    // ...но "первая" ЗДЕСЬ не освобождена — её буфер всё ещё жив в куче
    println!("{v}");                  // "вторая"
}   // ← вот ТУТ уничтожаются обе: сначала "вторая", потом "первая"
```

Уничтожение привязано к **выходу из блока** (области видимости), а не к перекрытию имени и не к тому, «остались ли ссылки». Это ключевой принцип: момент `Drop` определяется скоупом, а не подсчётом ссылок.

## 4. Перемещение (move) — поведение по умолчанию

Вот главное отличие от C++, и тут чаще всего спотыкаются. Что делает присваивание для владеющего типа?

```rust
let a = String::from("привет");
let b = a;              // НЕ копия! Владение ПЕРЕМЕЩ�ается из a в b
println!("{b}");        // ок
println!("{a}");        // ОШИБКА компиляции: значение из a перемещено
```

```text
error[E0382]: borrow of moved value: `a`
```

Что произошло физически: трёхсловная «шапка» (указатель, длина, ёмкость) скопировалась из `a` в `b`, а буфер в куче остался один. Если бы `a` осталась использоваться, мы получили бы **два владельца одного буфера** — и при выходе из блока буфер освободился бы дважды (double-free). Поэтому Rust, верный правилу «владелец один», **аннулирует `a`**: после `let b = a` старая переменная помечена как перемещённая, и обращаться к ней нельзя. Компилятор отслеживает это статически, без всякого флага в рантайме.

**Сравни с C++ и прочувствуй инверсию.** В C++ `b = a` по умолчанию **копирует** (зовётся copy-конструктор, буфер дублируется), а перемещение надо запросить явно через `std::move`. В Rust наоборот: **move — это поведение по умолчанию**, а копирование — то, что нужно запрашивать. C++ выбрал «безопасно, но иногда дорого по умолчанию»; Rust выбрал «дёшево и без дублей по умолчанию, а дублирование — осознанно».

## 5. Copy-типы: когда присваивание всё-таки копирует

Не всё перемещается. Простые значения, целиком лежащие на стеке и не владеющие ничем в куче, при присваивании **копируются**, и старая переменная остаётся жива:

```rust
let x = 5;
let y = x;          // копия: i32 — это просто 4 байта на стеке
println!("{x} {y}"); // ок, обе доступны — никакого move не было
```

Такие типы реализуют трейт `Copy`. Копировать их — это побитовое дублирование пары байт, дёшево, и у них нет деструктора (освобождать нечего), так что double-free невозможен в принципе. Поэтому Rust разрешает им неявное копирование.

`Copy` есть у:

1. Всех целых и дробных (`i32`, `u64`, `f64`, ...), `bool`, `char`.
2. Кортежей и массивов, **если все их элементы** `Copy` (например `(i32, bool)`).
3. Разделяемых ссылок `&T` (про них — в [`borrowing.md`](./borrowing.md)).

А вот `String`, `Vec<T>`, `Box<T>` — **не** `Copy`: они владеют кучей, поэтому перемещаются, а не копируются. Простое правило: **владеет ресурсом в куче → move; целиком на стеке и дёшев → copy.**

## 6. `clone` — явная глубокая копия

Если тебе реально нужны два независимых владеющих значения (а не перемещение), проси копию явно — методом `.clone()`:

```rust
let a = String::from("привет");
let b = a.clone();       // ГЛУБОКАЯ копия: буфер в куче продублирован
println!("{a} {b}");     // обе живы и независимы — у каждой свой буфер
```

Ключевая идея: дорогая операция (дублирование кучи) **видна в коде**. В C++ copy-конструктор вызывается незаметно, и можно случайно наделать дорогих копий в цикле, не заметив. В Rust `.clone()` всегда написан явно — стоимость на виду, и её легко отследить глазами и через `clippy`.

## 7. Передача в функции и возврат

Передача аргумента подчиняется тем же правилам: для владеющего типа вызов функции **перемещает** значение внутрь. Функция становится владельцем и по выходу его уничтожит:

```rust
fn consume(s: String) {       // s стала владельцем переданного значения
    println!("{s}");
}   // ← здесь переданная строка уничтожается

fn main() {
    let text = String::from("привет");
    consume(text);            // владение text перемещено в функцию
    // println!("{text}");    // ОШИБКА: text перемещён в consume и больше недоступен
}
```

Вернуть владение обратно можно через `return` — тогда значение перемещается наружу, вызывающему:

```rust
fn make() -> String {
    let s = String::from("создано тут");
    s                          // владение перемещается наружу, вызывающему
}                              // s НЕ уничтожается — она «уехала» в return
```

И тут вылезает неудобство, ради которого существует **заимствование**: если функции надо просто *посмотреть* на строку (например, посчитать длину), отдавать ей владение и получать обратно — абсурдно громоздко. Решение — давать функции временную **ссылку** вместо владения. Это следующий урок: [`borrowing.md`](./borrowing.md).

## Упражнение

Без проекта, в песочнице (`fn main`). Цель — поймать ошибки компилятора, а не написать рабочий код:

1. Заведи `let s1 = String::from("hi");`, потом `let s2 = s1;`. Попробуй напечатать `s1` — прочитай ошибку про moved value, осознай её.
2. Почини двумя способами по очереди: (а) через `.clone()`; (б) поменяв порядок так, чтобы `s1` больше не использовалась после перемещения.
3. Повтори опыт с `let a = 5; let b = a;` и убедись, что для `i32` ошибки **нет** — объясни себе, почему (`Copy`).
4. Напиши `fn len_of(s: String) -> usize { s.len() }`, вызови её и попробуй после вызова снова обратиться к строке. Получи ошибку. Подумай, насколько коряво было бы возвращать строку обратно только чтобы её переиспользовать, — это мотивация для `borrowing.md`.

## Шпаргалка

- **Три правила:** у значения один владелец; владелец вышел из скоупа → значение уничтожено; владелец всегда один.
- **`Drop`** = деструктор по выходу из блока (RAII); порядок — обратный объявлению. Момент уничтожения определяет **скоуп**, а не число ссылок.
- **Shadowing не уничтожает старое значение раньше** — оно доживает до конца блока, просто скрыто по имени.
- **Move — поведение по умолчанию** для владеющих типов (`String`, `Vec`, `Box`): присваивание/передача перемещает, источник аннулируется. Обратно к C++, где по умолчанию копия.
- **`Copy`-типы** (числа, `bool`, `char`, `&T`, кортежи/массивы из `Copy`) при присваивании копируются — источник остаётся жив.
- **`.clone()`** — явная глубокая копия; стоимость видна в коде.
- Передача в функцию перемещает владение; возврат перемещает его наружу.
- «Просто посмотреть, не забирая владение» → ссылки, см. [`borrowing.md`](./borrowing.md).
