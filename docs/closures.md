# Замыкания (closures)

> Ступень 5 · Урок C, закрывает ступень. Опирается на [`iterators.md`](./iterators.md), [`ownership.md`](./ownership.md), [`traits.md`](./traits.md).

В `iterators.md` мы писали `|x| x * 2` и читали как «функция на лету». Пора разобраться по-настоящему. Замыкание — это **анонимная функция, которая может захватывать переменные из окружающего кода**. Ты знаешь их по C++ (лямбды) и Python (вложенные функции/`lambda`). В Rust они устроены интереснее всего — потому что захват переменных подчиняется владению и заимствованию.

---

## 1. Синтаксис

```rust
let add_one = |x| x + 1;          // |параметры| выражение
let sum = |a, b| a + b;
let result = add_one(5);          // вызывается как функция → 6

// с телом в скобках и явными типами (типы обычно выводятся, можно опускать)
let describe = |x: i32| -> String {
    format!("число {x}")
};
```

Отличие от обычной `fn`: параметры в `|...|`, типы чаще всего **выводятся** из контекста (в отличие от функций, где они обязательны). Для C++-ника это лямбда `[](auto x){ return x+1; }`, для Python — `lambda x: x+1`, только мощнее.

---

## 2. Захват окружения — вот ради чего всё

Главное умение замыкания: оно **видит переменные из места, где объявлено**, без передачи их в параметры:

```rust
let factor = 3;
let multiply = |x| x * factor;    // factor НЕ параметр — захвачен из окружения
println!("{}", multiply(10));     // 30
```

`factor` не передан в `multiply` — замыкание «дотянулось» до него само. Обычная `fn` так не умеет: функция видит только свои аргументы и глобальные, но не локальные переменные вокруг. Это и есть разница между «функцией» и «замыканием».

И вот тут начинается специфика Rust: **как именно замыкание захватывает `factor`** — по ссылке или забирая владение? Ответ: по минимально необходимому. Компилятор смотрит, что замыкание делает с переменной, и выбирает самый «лёгкий» способ из трёх.

---

## 3. Три способа захвата = три знакомых режима владения

Захват подчиняется ровно той же тройке, что и всё в Rust ([`ownership.md`](./ownership.md), [`borrowing.md`](./borrowing.md)):

1. **По разделяемой ссылке `&T`** — если замыкание только **читает** переменную:

```rust
let s = String::from("привет");
let print_it = || println!("{s}");   // только читает s → захват по &
print_it();
println!("{s}");                      // s ещё доступна — её лишь одолжили
```

2. **По изменяемой ссылке `&mut T`** — если замыкание **меняет** переменную:

```rust
let mut count = 0;
let mut inc = || count += 1;   // меняет count → захват по &mut (и само замыкание mut)
inc();
inc();
println!("{count}");           // 2
```

3. **По владению (move)** — если замыкание **забирает** значение себе. Это запрашивают ключевым словом `move` (или компилятор делает сам, когда иначе нельзя):

```rust
let s = String::from("привет");
let consume = move || println!("{s}");   // move: s ПЕРЕМЕЩЕНА в замыкание
consume();
// println!("{s}");   // ОШИБКА: s больше не наша, она уехала в замыкание
```

То есть никакой новой модели: захват — это те же `&` / `&mut` / move, что ты уже знаешь, просто применённые к переменным из окружения.

---

## 4. Зачем `move`: переживание окружения

Главный практический повод для `move` — когда замыкание должно **пережить** область, где создано. Классика — отдать замыкание в поток (Ступень 7) или вернуть из функции. Если бы оно захватило по ссылке, ссылка повисла бы, когда исходный скоуп умрёт (привет, времена жизни из [`lifetimes.md`](./lifetimes.md)). `move` решает это, перенося данные внутрь замыкания:

```rust
fn make_printer() -> impl Fn() {
    let s = String::from("я переживу функцию");
    move || println!("{s}")     // без move: ссылка на s повисла бы после возврата
}                               // с move: s уехала в замыкание и живёт вместе с ним
```

Это ровно та же логика, что в `borrowing.md` про висячие ссылки — `move` переносит владение, чтобы заёма-долгожителя не возникло.

---

## 5. `Fn`, `FnMut`, `FnOnce` — замыкания это тоже трейты

Откуда компилятор знает, какое замыкание куда годится? Снова **трейты** (Ступень 4). Каждое замыкание автоматически реализует один или несколько из трёх, по тому, **как оно обращается с захваченным**:

1. **`Fn`** — только читает захваченное (`&`). Можно звать много раз, ничего не портя.
2. **`FnMut`** — меняет захваченное (`&mut`). Тоже много раз, но требует `mut`.
3. **`FnOnce`** — потребляет захваченное (move). Можно вызвать **один раз** (после первого вызова захваченное израсходовано).

Они вложены: всякое `Fn` годится как `FnMut`, всякое `FnMut` — как `FnOnce`. Когда функция принимает замыкание, она в сигнатуре указывает, что ей достаточно:

```rust
fn call_twice<F: Fn()>(f: F) {   // требует Fn — значит f можно звать многократно
    f();
    f();
}
```

Вот теперь видно, что происходило в `iterators.md`: `map(|x| x * 2)` принимает замыкание с bound'ом `FnMut`, потому применяет его к каждому элементу. Адаптеры итераторов — это обобщённые функции, берущие замыкания по этим трейтам. Всё сходится в одну систему.

---

## 6. Замыкание vs обычная функция: почему два механизма

Если замыкание ничего не захватывает, оно эквивалентно функции, и туда, где ждут замыкание, можно передать обычную `fn`:

```rust
let v = vec![1, 2, 3];
let doubled: Vec<i32> = v.iter().map(|x| x * 2).collect();   // замыкание
fn triple(x: &i32) -> i32 { x * 3 }
let tripled: Vec<i32> = v.iter().map(triple).collect();       // обычная fn тоже подходит
```

Разница нужна именно для **захвата**: когда логика зависит от локального состояния (`factor`, `count`, `s` из примеров), функция бессильна — ей нечем дотянуться до окружения, а замыкание захватывает. Это и есть его смысл существования.

---

## Упражнение

1. Заведи `let n = 10;` и замыкание `|x| x + n`, захватывающее `n`. Вызови. Убедись, что `n` после вызова доступна (захват по `&`).
2. Заведи `let mut total = 0;` и замыкание, прибавляющее аргумент к `total`. Вызови дважды, выведи `total`. Не забудь `mut` у замыкания — поймай ошибку, если забудешь.
3. Возьми `String`, создай `move`-замыкание, печатающее её, вызови, потом попробуй обратиться к строке снаружи — поймай ошибку про move. Объясни себе, что захватилось.
4. Напиши `fn apply<F: Fn(i32) -> i32>(f: F, x: i32) -> i32 { f(x) }` и передай в неё разные замыкания. Затем поменяй bound на `FnMut`/`FnOnce` и посмотри, что меняется.
5. Свяжи с итераторами: отфильтруй `Vec<i32>` по порогу, заданному **внешней переменной** `threshold`, через `.filter(|x| **x > threshold)`. Это и есть захват в действии — обычной `fn` так не написать.

---

## Шпаргалка

- **Замыкание** = анонимная функция `|args| body`, умеющая **захватывать** переменные окружения (чего `fn` не может).
- Типы параметров обычно **выводятся** (в отличие от `fn`).
- **Захват = та же тройка владения:** по `&` (только читает), по `&mut` (меняет), по **move** (забирает; запрашивается словом `move`). Компилятор берёт минимально необходимое.
- **`move`** нужен, когда замыкание должно пережить свой скоуп (поток, возврат из функции) — иначе захваченная ссылка повисла бы.
- **Три трейта:** **`Fn`** (читает, много раз), **`FnMut`** (меняет, много раз), **`FnOnce`** (потребляет, один раз). Вложены: `Fn` ⊂ `FnMut` ⊂ `FnOnce`.
- Функции принимают замыкания по bound'у (`F: Fn(...)`) — это обобщения из [`generics.md`](./generics.md); так и устроены `map`/`filter`.
- Не захватываешь ничего → можно передать обычную `fn` вместо замыкания.
