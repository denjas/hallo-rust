# Обобщения (generics)

> Ступень 4 · Урок A. Парный файл: [`traits.md`](./traits.md). Опирается на `<T>` из [`enums.md`](./enums.md) (`Option<T>`, `Result<T, E>`).

Ты уже видел `<T>` — в `Option<T>` и `Result<T, E>`. Это и есть обобщения: способ написать код **один раз** для любого типа, вместо копипасты под каждый. Для тебя это знакомо по двум сторонам: шаблоны C++ и «утиная типизация» Python. Rust берёт мощь первых и убирает их главные болячки.

---

## 1. Проблема, которую решают обобщения

Допустим, нужна функция «наибольший элемент среза». Без обобщений пришлось бы писать её отдельно под `i32`, под `f64`, под каждый тип — один и тот же код, разные сигнатуры. Обобщение пишет его раз:

```rust
fn largest<T>(list: &[T]) -> &T {     // T — параметр-тип, как <T> в Option<T>
    let mut biggest = &list[0];
    for item in list {
        if item > biggest {           // ← вот здесь будет загвоздка, см. ниже
            biggest = item;
        }
    }
    biggest
}
```

`<T>` после имени объявляет **параметр-тип** (ровно как `<'a>` объявлял время жизни в [`lifetimes.md`](./lifetimes.md)). Дальше `T` используется как обычный тип: `&[T]` — срез из `T`, `&T` — ссылка на `T`.

Но этот код пока **не компилируется**, и причина важна — она и есть главное отличие Rust от C++.

---

## 2. Главное отличие от шаблонов C++: ограничения (bounds)

В строке `if item > biggest` мы сравниваем два `T`. Но `T` — это «любой тип», а **не любой тип вообще умеет сравниваться через `>`**. Компилятор Rust отказывается принимать функцию на веру:

```text
error[E0369]: binary operation `>` cannot be applied to type `&T`
```

И вот ключевая разница. В C++ шаблон проверяется **в момент инстанцирования**: ты подставил конкретный тип, и если он не умеет `>`, компилятор вываливает простыню ошибок где-то в недрах шаблона. В Rust наоборот: обобщённая функция проверяется **в момент объявления**. Чтобы пользоваться `>`, ты обязан заранее **пообещать**, что `T` это умеет — через **ограничение** (trait bound):

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {   // T: PartialOrd — "T умеет сравниваться"
    let mut biggest = &list[0];
    for item in list {
        if item > biggest {
            biggest = item;
        }
    }
    biggest
}
```

`T: PartialOrd` читается «T реализует трейт `PartialOrd`» (трейт сравнения; подробно трейты — в [`traits.md`](./traits.md)). Теперь компилятор знает, что `>` для `T` законен, и функция собирается.

Что это даёт по сравнению с C++:

1. **Ошибка ловится в объявлении, а не в инстанцировании.** Если тело использует то, чего bound не обещает, — ошибка прямо тут, в одном месте, понятным сообщением. Не нужно подставлять тип, чтобы узнать, что шаблон сломан.
2. **Ошибка на стороне вызова — короткая и ясная.** Передал тип без `PartialOrd` → «тип не реализует `PartialOrd`», а не каскад ошибок из глубины шаблона.
3. **Сигнатура — это контракт.** По `<T: PartialOrd>` сразу видно, что функция требует от типа. В C++ требования шаблона неявны и разбросаны по телу (до C++20 с концептами).

Это та же философия, что у времён жизни: вместо «доверяй и подставляй» — «опиши требования заранее, компилятор проверит».

---

## 3. А как же утиная типизация Python

В Python ты бы написал ту же функцию без всяких объявлений: `if item > biggest` просто сработает, если у объекта есть `__gt__`, и рухнет в рантайме, если нет. Это утиная типизация: «крякает как утка — сойдёт за утку», проверка откладывается до запуска.

Rust даёт тот же результат («работает с любым сравнимым типом»), но проверку двигает на этап компиляции. `T: PartialOrd` — это как если бы Python заранее, до запуска, гарантировал, что у переданного объекта есть `__gt__`. Получаешь гибкость утиной типизации **без** её риска «упадёт в проде на редком типе».

---

## 4. Обобщения бесплатны в рантайме: мономорфизация

Главный вопрос производительности: во что это компилируется? В **мономорфизацию** — то же, что делают шаблоны C++. Для каждого конкретного типа, с которым ты реально вызвал обобщённую функцию, компилятор генерирует **отдельную специализированную копию**:

```rust
let ints = vec![3, 7, 2];
let floats = vec![1.5, 0.2];
largest(&ints);     // компилятор сгенерирует largest для i32
largest(&floats);   // и отдельную копию largest для f64
```

В итоговом бинарнике это две разные функции, каждая заточена под свой тип, без всякой диспетчеризации в рантайме. **Обобщения не стоят ничего во время выполнения** — никаких проверок типа, никакого бокса. Платишь только размером бинарника (копии кода) и временем компиляции. Это «zero-cost abstraction»: абстракция есть в исходнике, но испаряется в машинном коде. (Есть и второй способ — динамическая диспетчеризация через `dyn Trait`, с рантайм-стоимостью; разберём в [`traits.md`](./traits.md).)

## 5. Обобщения в структурах, enum'ах и методах

`<T>` работает не только в функциях. Структуры и enum'ы параметризуются так же — ты это уже видел в `Option<T>`:

```rust
struct Point<T> {
    x: T,
    y: T,
}

enum Option<T> {        // тот самый Option из стандартной библиотеки
    Some(T),
    None,
}
```

Методы для обобщённой структуры объявляют с `impl<T>`:

```rust
impl<T> Point<T> {
    fn x(&self) -> &T {
        &self.x
    }
}
```

`impl<T>` читается «для любого `T`, вот методы `Point<T>`». А можно написать `impl` только для конкретного типа — тогда метод появится лишь у `Point<f64>`, но не у других:

```rust
impl Point<f64> {
    fn dist_from_origin(&self) -> f64 {   // есть ТОЛЬКО у Point<f64>
        (self.x * self.x + self.y * self.y).sqrt()
    }
}
```

Несколько параметров — через запятую, как у `Result<T, E>`: `struct Pair<T, U> { a: T, b: U }`.

## 6. Несколько ограничений сразу

Типу можно навесить несколько требований — через `+`, либо вынести в `where` для читаемости:

```rust
fn show_largest<T: PartialOrd + std::fmt::Display>(list: &[T]) {
    // T умеет и сравниваться (PartialOrd), и печататься (Display)
    println!("наибольший: {}", largest(list));
}

// то же самое, но через where — удобно, когда ограничений много
fn show_largest<T>(list: &[T])
where
    T: PartialOrd + std::fmt::Display,
{
    println!("наибольший: {}", largest(list));
}
```

`PartialOrd + Display` = «и сравнимый, и печатаемый». Это и есть мостик к следующему уроку: ограничения — это всегда **трейты**, а трейты — отдельная большая тема. Что такое трейт, как его объявить и реализовать — в [`traits.md`](./traits.md).

## Упражнение

1. Напиши `largest` без bound'а и поймай `E0369` (бинарная операция `>` неприменима). Прочитай — это ровно «ошибка в объявлении, а не в инстанцировании», фирменное отличие от C++.
2. Добавь `T: PartialOrd`, добейся компиляции, вызови на `Vec<i32>` и `Vec<f64>`.
3. Объяви `struct Pair<T> { a: T, b: T }` и метод `impl<T: PartialOrd> Pair<T>` с функцией `larger(&self) -> &T`, возвращающей больший из двух.
4. Добавь `impl Pair<i32>` с методом, которого нет у других `Pair` (например `sum(&self) -> i32`). Убедись, что на `Pair<f64>` этот метод недоступен — и объясни себе, почему так бывает полезно.

## Шпаргалка

- **`<T>`** — параметр-тип: пишешь код раз, работает для любого типа (как `Option<T>`).
- **Обязательно объявить `<T>`** после имени (`fn f<T>`, `struct S<T>`, `impl<T>`), как `<'a>` для сроков.
- **Bound `T: Trait`** — обещание, что `T` умеет нужное; без него нельзя использовать соответствующие операции. Несколько — через `+` или `where`.
- **Отличие от C++:** ошибка ловится в объявлении функции, а не при инстанцировании; сигнатура — явный контракт.
- **Отличие от Python:** та же гибкость «любой подходящий тип», но проверка на компиляции, а не падение в рантайме.
- **Мономорфизация:** компилятор генерирует отдельную копию под каждый реальный тип → **ноль стоимости в рантайме** (как шаблоны C++), ценой размера бинарника.
- Структуры/enum'ы/методы параметризуются так же: `struct P<T>`, `impl<T> P<T>`, либо `impl P<КонкретныйТип>` для типо-специфичных методов.
- Все bound'ы — это **трейты** → [`traits.md`](./traits.md).
