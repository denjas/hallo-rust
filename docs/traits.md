# Трейты (traits)

> Ступень 4 · Урок B, закрывает ступень. Опирается на [`generics.md`](./generics.md) (bound'ы — это и есть трейты).

В `generics.md` мы писали `T: PartialOrd` — «T умеет сравниваться». `PartialOrd` — это **трейт**. Трейт — это **набор поведения, который тип обещает предоставить**: список методов, которые тип обязуется реализовать. Ты уже встречал трейты, не зная имени: `#[derive(Debug, Clone, PartialEq)]` из ранних уроков добавлял типу реализации трейтов `Debug`, `Clone`, `PartialEq`.

Для C++-головы трейт — это смесь чистого абстрактного класса (интерфейса) и концептов C++20. Но устроен он иначе, и эти отличия — самое важное в уроке.

---

## 1. Объявление и реализация

Трейт описывает, **что** тип должен уметь, не говоря **как**:

```rust
trait Summary {
    fn summarize(&self) -> String;     // сигнатура без тела — обязателен к реализации
}
```

Реализуют трейт для конкретного типа отдельным блоком `impl Trait for Type`:

```rust
struct Article {
    title: String,
    body: String,
}

impl Summary for Article {            // "реализуем Summary для Article"
    fn summarize(&self) -> String {
        format!("{}: {}…", self.title, &self.body[..20])
    }
}
```

Теперь у `Article` есть метод `summarize`, и — что важнее — `Article` годится везде, где требуется «что-то, реализующее `Summary`».

**Первое отличие от C++.** Связь «тип ↔ трейт» объявляется **отдельно** от самого типа. В C++ интерфейс ты обязан указать в объявлении класса (`class Article : public Summary`). В Rust `impl Summary for Article` пишется где угодно, и тип о трейте «узнаёт» постфактум. Из этого следует мощная штука: **ты можешь реализовать свой трейт для чужого типа** — например, добавить `Summary` стандартному `String` или типу из чужой библиотеки. В C++/Java так нельзя: не можешь же ты вписать свой интерфейс в чужой готовый класс. (Ограничение одно — «правило сироты»: в твоём крейте трейт или тип должен быть твоим, чтобы не было хаоса с конфликтующими реализациями.)

## 2. Методы по умолчанию

Трейт может давать методам **тело по умолчанию** — тогда тип-реализатор получает их даром и при желании переопределяет:

```rust
trait Summary {
    fn summarize(&self) -> String;            // обязателен

    fn preview(&self) -> String {             // метод по умолчанию
        format!("Читать далее: {}", self.summarize())   // может звать другие методы трейта
    }
}
```

Реализуешь только `summarize` — `preview` приедет бесплатно. Это покрывает значимую часть того, ради чего в C++ заводят наследование с общими методами в базовом классе, но без самого наследования.

## 3. Две стороны использования трейта

Трейт можно требовать двумя способами, и разница между ними — ось всего урока: **статическая диспетчеризация против динамической.**

### Статическая: `impl Trait` / bound (то же, что в generics.md)

```rust
fn notify(item: &impl Summary) {       // "любой тип, реализующий Summary"
    println!("{}", item.summarize());
}
// эквивалентная запись через bound:
fn notify<T: Summary>(item: &T) { ... }
```

Это обобщение из прошлого урока: компилятор **мономорфизирует** — генерирует под каждый конкретный тип свою копию `notify`. Вызов метода известен на этапе компиляции, диспетчеризации в рантайме нет. Быстро, ноль стоимости — но тип на каждом вызове фиксирован.

### Динамическая: `dyn Trait` — трейт-объект

А что если в одном `Vec` надо хранить **разные** типы, объединённые трейтом? Мономорфизация тут не годится (тип-то не один). Нужен **трейт-объект** — `dyn Trait` за ссылкой или `Box`:

```rust
let items: Vec<Box<dyn Summary>> = vec![     // вектор РАЗНЫХ типов, всех с Summary
    Box::new(article),
    Box::new(tweet),
];
for item in &items {
    println!("{}", item.summarize());        // какой именно summarize — решается в РАНТАЙМЕ
}
```

`dyn Summary` означает «какой-то тип, реализующий `Summary`, конкретика неизвестна до запуска». Вызов `summarize` идёт через **таблицу виртуальных методов** (vtable) — ровно как виртуальные функции C++. Это и есть динамическая диспетчеризация.

### Чем платишь за `dyn`

1. **Косвенный вызов** через vtable — чуть медленнее прямого, и компилятор не может его заинлайнить.
2. **Указатель «толще»** — трейт-объект это (указатель на данные + указатель на vtable), «жирный указатель».
3. Зато **гибкость**: разнородные типы в одной коллекции, и меньше раздувания бинарника (одна копия кода вместо копии под каждый тип).

Правило выбора: **`impl Trait`/generics по умолчанию** (быстро, статически); **`dyn Trait`, когда нужна гетерогенная коллекция или когда тип принципиально известен только в рантайме.** В C++ ты этот выбор делаешь между шаблоном и виртуальной функцией — здесь ровно та же развилка, только явная в синтаксисе (`impl`/`<T>` vs `dyn`).

## 4. derive: трейты, которые пишет компилятор

Возврат к `#[derive(...)]`, который ты видел с первых уроков. Для распространённых трейтов компилятор умеет генерировать реализацию сам:

```rust
#[derive(Debug, Clone, PartialEq)]
struct Point {
    x: f64,
    y: f64,
}
```

`#[derive(Debug)]` пишет реализацию `Debug` (печать через `{:?}`), `Clone` — метод `.clone()`, `PartialEq` — оператор `==`. Это просто автогенерация того самого `impl Trait for Type`, которую тебе не пришлось писать руками для очевидных случаев. Под капотом — обычные трейты, никакой магии.

## 5. Где трейты сходятся со всем курсом

Трейты — это объединяющий механизм Rust, и теперь видно, как ранние уроки в них упираются:

1. **Bound'ы обобщений** (`generics.md`) — это требования трейтов: `T: PartialOrd`.
2. **Операторы** — это трейты: `+` это `Add`, `==` это `PartialEq`, `<` это `PartialOrd`. Реализуешь трейт — получаешь оператор для своего типа.
3. **`Drop`** из [`ownership.md`](./ownership.md) — трейт: реализуя его, ты задаёшь, что происходит при уничтожении значения.
4. **`Display`/`Debug`** — трейты форматирования (`{}` и `{:?}` из [`basics.md`](./basics.md)).
5. **Итераторы** (Ступень 5) и **многопоточность** (`Send`/`Sync`, Ступень 7) — тоже целиком на трейтах.

Вместо наследования (которого в Rust нет вовсе) переиспользование поведения идёт через трейты и композицию. Это сознательный выбор: «состав из способностей», а не «дерево классов».

## Упражнение

1. Объяви `trait Greet { fn hello(&self) -> String; }`. Реализуй его для двух своих структур (`Dog`, `Cat`) по-разному.
2. Добавь в `Greet` метод по умолчанию `fn greet_loudly(&self) -> String`, возвращающий `hello()` в верхнем регистре (`.to_uppercase()`). Реализаторам его дописывать не нужно — проверь, что он есть у обоих.
3. Напиши `fn announce(g: &impl Greet)` — статическая диспетчеризация. Вызови на `Dog` и `Cat`.
4. Сложи оба в `Vec<Box<dyn Greet>>` и пройди циклом, зовя `hello()`. Прочувствуй разницу: п.3 мономорфизируется под каждый тип, а п.4 диспетчеризуется в рантайме через vtable.
5. Бонус: реализуй свой трейт для **чужого** типа — например `impl Greet for String`. Убедись, что это легально (твой трейт), и подумай, почему `impl Display for String` тебе бы уже не разрешили (правило сироты).

## Шпаргалка

- **Трейт** = набор поведения (методов), которое тип обещает предоставить; объявляется `trait`, реализуется `impl Trait for Type`.
- **Реализация объявляется отдельно от типа** → можно реализовать **свой** трейт для **чужого** типа (с оглядкой на правило сироты). В C++/Java так нельзя.
- **Методы по умолчанию** — тело в трейте; реализатор получает даром, может переопределить. Замена общим методам базового класса.
- **`impl Trait` / `<T: Trait>`** — статическая диспетчеризация, мономорфизация, ноль стоимости, тип фиксирован на вызове.
- **`dyn Trait`** (за `&`/`Box`) — динамическая диспетчеризация через vtable; для разнородных коллекций; цена — косвенный вызов и жирный указатель.
- Правило выбора: generics по умолчанию, `dyn` — когда нужна гетерогенность или тип известен лишь в рантайме (как шаблон vs virtual в C++).
- **`#[derive(...)]`** — автогенерация `impl` для частых трейтов (`Debug`, `Clone`, `PartialEq`, ...).
- Трейты пронизывают весь язык: операторы, `Drop`, `Display`/`Debug`, итераторы, `Send`/`Sync`. Наследования нет — переиспользование через трейты и композицию.
