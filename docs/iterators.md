# Итераторы (iterators)

> Ступень 5 · Урок B. Опирается на [`collections.md`](./collections.md), [`traits.md`](./traits.md). Парный файл: [`closures.md`](./closures.md).

Если ты любил в Python `map`/`filter`, генераторы и comprehensions — здесь почти то же самое, только на этапе компиляции и без рантайм-стоимости. Итераторы — один из самых приятных инструментов Rust: цепочки преобразований данных, которые читаются как описание намерения, а компилируются в такой же быстрый код, как ручной цикл.

---

## 1. Итератор — это трейт

Под капотом итератор — реализация трейта `Iterator` (вот зачем была Ступень 4):

```rust
trait Iterator {
    type Item;                          // ассоциированный тип: что выдаём
    fn next(&mut self) -> Option<Self::Item>;   // следующий элемент, или None в конце
}
```

Весь итератор — это один метод `next`, который выдаёт `Some(элемент)`, пока они есть, и `None`, когда кончились. Знакомая пара из [`enums.md`](./enums.md): `Option` тут кодирует «есть ещё / всё». Цикл `for` — это просто сахар над повторным вызовом `next` до `None`:

```rust
for x in &v { ... }
// разворачивается примерно в: пока next() даёт Some(x) — выполняй тело
```

Три способа получить итератор по коллекции — и это ровно тройка владения из [`borrowing.md`](./borrowing.md):

1. `v.iter()` — выдаёт `&T` (заимствование на чтение).
2. `v.iter_mut()` — выдаёт `&mut T` (заимствование на изменение).
3. `v.into_iter()` — выдаёт `T`, **поглощая** коллекцию (move).

`for x in &v` зовёт `.iter()`, `for x in &mut v` → `.iter_mut()`, `for x in v` → `.into_iter()`. Та самая тройка из прошлого урока, теперь явно.

## 2. Ленивость — ключевая идея

Самое важное про итераторы: **адаптеры ленивы. Они ничего не делают, пока у них не запросят результат.** Это как генераторы Python: `map(...)` сам по себе не считает — он описывает, *что* считать.

```rust
let v = vec![1, 2, 3, 4, 5];

let it = v.iter().map(|x| x * 2);   // НИЧЕГО не вычислено! map только описал намерение
// никакого удвоения ещё не произошло
```

`map(|x| x * 2)` не пробежал по вектору — он вернул новый итератор, который *при запросе* будет удваивать. Замыкание `|x| x * 2` (синтаксис `|аргумент| тело`) — про них отдельный урок [`closures.md`](./closures.md); пока читай как «функция на лету». Вычисление запускается только когда цепочку **потребляют** (раздел 4).

Зачем лень: можно строить длинные цепочки преобразований, и они выполнятся **за один проход**, без создания промежуточных коллекций на каждом шаге. `filter().map().take(3)` не делает три прохода и три временных вектора — он тянет элементы по одному через всю цепочку.

## 3. Адаптеры — преобразования (ленивые, возвращают итератор)

Адаптер берёт итератор и возвращает новый итератор. Их можно цепочкой. Самые ходовые:

```rust
let nums = vec![1, 2, 3, 4, 5, 6];

nums.iter()
    .map(|x| x * x)          // преобразовать каждый: 1,4,9,16,25,36
    .filter(|x| *x % 2 == 0) // оставить по предикату: 4,16,36
    .take(2)                 // взять первые N: 4,16
    .skip(1);                // пропустить первые N
// всё это ЛЕНИВО — пока ничего не посчитано
```

1. `map(f)` — применить `f` к каждому элементу.
2. `filter(pred)` — оставить элементы, где `pred` истинно.
3. `take(n)` / `skip(n)` — взять / пропустить первые `n`.
4. `enumerate()` — выдавать `(индекс, элемент)`, как `enumerate` в Python.
5. `zip(other)` — склеить два итератора в пары, как `zip` в Python.
6. `rev()` — обратный порядок.

Это прямые аналоги Python'овских `map`/`filter`/`enumerate`/`zip`, только ленивые по умолчанию и без рантайм-накладных (см. §5).

## 4. Потребители — запускают работу (возвращают значение)

Чтобы лень превратилась в результат, цепочку надо **потребить**. Потребитель проходит итератор до конца и возвращает что-то не-итераторное:

```rust
let nums = vec![1, 2, 3, 4, 5];

let sum: i32 = nums.iter().sum();                  // сумма
let count = nums.iter().filter(|x| **x > 2).count(); // сколько штук
let doubled: Vec<i32> = nums.iter().map(|x| x * 2).collect(); // СОБРАТЬ в коллекцию
let found = nums.iter().find(|x| **x == 3);        // первый подходящий → Option<&i32>
let any_big = nums.iter().any(|x| *x > 4);         // есть ли хоть один → bool
```

Главный потребитель — **`collect()`**: прогоняет всю ленивую цепочку и собирает результат в коллекцию (`Vec`, `HashMap`, `String` — куда скажешь типом). Вот тут наконец происходит вся работа:

```rust
// классическая цепочка: взять чётные, удвоить, собрать в Vec
let result: Vec<i32> = (1..=10)              // диапазон — тоже итератор!
    .filter(|x| x % 2 == 0)
    .map(|x| x * 2)
    .collect();                              // ← здесь всё и выполняется: [4, 8, 12, 16, 20]
```

Обрати внимание на `: Vec<i32>` — `collect` смотрит на тип, в который ты собираешь, и решает, что строить. Собери в `String` — склеит символы; в `HashMap` — построит словарь из пар. Это замена Python'овским comprehensions: `[x*2 for x in range(1,11) if x%2==0]` ↔ цепочка выше.

Деталь про `**x` в примерах: `iter()` по `Vec<i32>` даёт `&i32`, а в `filter` замыкание получает ещё одну ссылку — отсюда двойное разыменование. Это поначалу раздражает; на практике часто чинится через `copied()`/`cloned()` в цепочке. Не зацикливайся, привыкнется.

## 5. Почему это так же быстро, как ручной цикл

Главный страх человека из C++: «красивые цепочки = медленно?» Нет. Итераторы в Rust — **zero-cost abstraction**. Каждый адаптер — это обобщённый тип (из [`generics.md`](./generics.md)), и вся цепочка **мономорфизируется и инлайнится** компилятором в один проход без промежуточных аллокаций. Сгенерированный машинный код для `iter().map().filter().sum()` идентичен (или лучше) тому, что ты написал бы руками одним циклом.

Контраст с Python, где `map`/`filter` — это реальные объекты с вызовами на каждом шаге и заметными накладными. В Rust абстракция есть в исходнике и испаряется в бинарнике. Поэтому идиоматичный Rust поощряет итераторные цепочки даже в горячем коде — читаемость **бесплатна**.

## Упражнение

1. Из `vec![1,2,3,4,5,6]` получи `Vec` квадратов только нечётных чисел, через `.iter().filter().map().collect()`. Проверь тип результата аннотацией `: Vec<i32>`.
2. Посчитай сумму всех чисел от 1 до 100 одной строкой: `(1..=100).sum()`. (Диапазон — итератор.)
3. Возьми `Vec<&str>` слов и собери их в одну `String` через `.collect()` (подскажет тип `String`). Поэкспериментируй с `.rev()` перед сбором.
4. Докажи себе лень: напиши `let it = v.iter().map(|x| { println!("трогаю {x}"); x * 2 });` и убедись, что **до** потребления (`collect`/`for`) ничего не печатается. Печать пойдёт только когда добавишь `.collect::<Vec<_>>()`.
5. Перепиши обычный `for`-цикл с ручным счётчиком и условием в итераторную цепочку. Сравни читаемость.

## Шпаргалка

- **Итератор = трейт `Iterator`** с методом `next() -> Option<Item>`; `for` — сахар над ним.
- Получение: **`iter()`** (`&T`), **`iter_mut()`** (`&mut T`), **`into_iter()`** (`T`, поглощает) — тройка владения.
- **Адаптеры ленивы**: `map`, `filter`, `take`/`skip`, `enumerate`, `zip`, `rev` — возвращают итератор, ничего не считают.
- **Потребители запускают работу**: `collect`, `sum`, `count`, `find`, `any`/`all`, `fold`. `collect` строит коллекцию по типу-цели.
- Лень → длинные цепочки выполняются **за один проход**, без промежуточных коллекций.
- **Zero-cost**: цепочки мономорфизируются и инлайнятся в код уровня ручного цикла. В отличие от Python, читаемость бесплатна.
- Прямые аналоги Python: `map`/`filter`/`enumerate`/`zip` и comprehensions ↔ `…collect()`.
